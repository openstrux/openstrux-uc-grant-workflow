## Why

The grant workflow repository is initialized with a frontend and tests but no backend. The backend must be generated by either of two paths (direct or Openstrux-assisted) against the same baseline. This change defines exactly what must be built — it is the task that both paths execute.

## What Changes

### Prisma schema
- Complete `prisma/schema.prisma` with all P0-P2 models: `Submission`, `ProposalVersion`, `ApplicantIdentity`, `BlindedPacket`, `EligibilityRecord`, `AuditEvent`, `Call`
- Relations and indexes as required by the domain model

### Domain package (`packages/domain/`)
- Zod schemas for all entities in `packages/domain/src/schemas/`
- TypeScript types in `packages/domain/src/entities/`

### Policy package (`packages/policies/`)
- `packages/policies/src/eligibility/evaluateEligibility.ts` — pure rule evaluation
- `packages/policies/src/access/checkAccess.ts` — principal + resource + operation check
- `packages/policies/src/workflow/submitProposal.ts` — intake service
- `packages/policies/src/workflow/runEligibilityCheck.ts` — eligibility service

### API routes (`app/web/src/app/api/`)
- `app/web/src/app/api/intake/route.ts` — POST handler (replace stub)
- `app/web/src/app/api/eligibility/route.ts` — POST handler (replace stub)

### Access middleware
- `app/web/src/server/auth/middleware.ts` — JWT role extraction
- `app/web/src/server/auth/requireRole.ts` — role guard helper

## Acceptance criteria

All tests in `tests/unit/` and `tests/integration/` pass.
`tsc --noEmit` exits 0 in `app/web/`.
